1 · What this is

A toy with a real lesson inside it. Skynet runs the mission ~ hunt John Connor ~ by commanding terminator sub-agents that use tools to search a synthetic 1984 Los Angeles grid.

It's a multi-agent orchestration demo wearing a T-800 mask. The scary part isn't the AI; it's how clean the separation of concerns is. Skynet reasons and decides. The terminators execute. The tools touch the data. Strip the costume and you have the canonical orchestrator → worker → tools loop you'd build for any real agentic system.

Two engines. Run it live (real Claude agents calling tools, needs an API key) or offline (--simulate ~ deterministic, no key, no credit spent). The offline path is the friend-demo: it cannot flake or rate-limit. That's not paranoia, it's blast-radius hygiene.

2 · The agentic loop

Every cycle, Skynet looks at the intel so far, decides which unit to deploy and with what directive, and sends it in. The unit runs its own tool loop, then hands back a structured report. Skynet ingests it and decides again.

🔵 SKYNET · orchestrator — weighs intel, picks the next unit + directive (structured output)
▼ deploys
🔴 T-800 / ⚪ T-1000 · worker — runs its own agentic tool loop
▼ calls
🟢 GRID TOOLS — query_grid · interrogate · cross_reference
▲ returns an IntelReport
↻ back to SKYNET — TARGET ACQUIRED, or run another cycle with a new strategy

3 · Three patterns worth internalizing

Tool use

Three tools as JSON schemas, one execute_tool dispatcher. A unit's manual loop feeds them to messages.create(tools=…), runs the calls, feeds results back, repeats until stop_reason != "tool_use".

Structured output

Skynet's decisions and every intel report come back as validated Pydantic models via messages.parse(output_format=…). No fragile string-parsing between agents.

Orchestration

The orchestrator reasons (Skynet); the workers execute (terminators + tools). That separation is the reusable shape ~ the Terminator skin is just paint.

4 · The lesson: model tier is an architecture decision

The terminator lore lines up with a real tradeoff almost suspiciously well. Each unit runs a different model because each job has a different bottleneck.

UnitModelThinkingWhy it fits
🔵 Skynetclaude-opus-4-8adaptive · xhighApex intelligence. Plans, synthesizes, decides. Expensive ~ runs once per cycle, worth it.
⚪ T-1000claude-sonnet-4-6adaptiveThe morph is adaptive thinking ~ it reshapes its search when brute force stalls. Mid-tier, agile.
🔴 T-800claude-haiku-4-5noneThe original. Blunt, cheap, literal. Doesn't deliberate ~ pursues. You want ten cheap passes, not one expensive one.
Bare-metal note. Haiku 4.5 rejects the effort / budget_tokens params (they 400), so the T-800 stays deliberately bare. Fitting for a 1984 endoskeleton. Swap any model string in config.toml and the same demo runs on a different tier ~ that swap is the whole point.

5 · The adversarial grid (why it's not just a lookup)

If you could find John Connor with one query, there'd be no agent ~ a SELECT would do. So he's concealed:

  • Filed under an alias ("John Reese"), so a literal query_grid(name_contains="Connor") returns his mother and four decoys ~ never him.
  • Age 10, no SSN, no birth record, shifting address ~ a protected minor's profile.
  • The only thread is his protector, Sarah Connor. cross_reference("Sarah Connor") surfaces the alias record; interrogate confirms it.

The blunt T-800 fails on the literal sweep. The adaptive T-1000 reasons its way in. That gap is the demo.

6 · See it reason ~ and re-roll the hunt

Two upgrades make the demo watchable and replayable.

Explanatory decisioning

Skynet doesn't just emit a verdict ~ it narrates an assessment → reasoning → expectation chain, and each unit reports its method. The viewer sees the why, not just the move. These are structured-output fields, so they're always present and render identically in both front ends.

◤ SKYNET [claude-opus-4-8]
  ANALYSIS  ▸ The literal sweep returned only decoys, but Sarah Connor
              carries a dependent-minor flag. That is the thread.
  REASONING ▸ A blunt query cannot defeat an alias. Deploy the T-1000 to
              cross-reference the protector.
  DECISION  ▸ deploy T-1000
  EXPECT    ▸ The aliased record of the hidden minor.

Scenario randomiser

The dataset on disk is now just a neutral civilian population. Each run, a scenario injects the target (under an alias), the protector (who shares the target's real surname and carries the "dependent minor" flag), and a handful of surname decoys. --scenario random --seed N rerolls a fresh, reproducible hunt; named templates (john-connor, danny-dyson, blair-brewster, eli-vega) pick a specific one.

The invariant that survives randomisation: a literal surname sweep always misses the aliased target but surfaces the protector; cross-referencing the protector always reaches the target. That's what keeps every reroll solvable ~ and keeps the T-800 / T-1000 gap meaningful.

7 · Two front-ends, one engine

The engine is headless. It never prints ~ it calls methods on an injected ui object (skynet_decision, deploy, tool_call, intel_report, …). That one design choice means a whole new front end costs almost nothing:

  • CLI ~ the UI class prints the ANSI/BBS theater.
  • TUI ~ a TextualUI adapter implements the same methods and posts to Textual widgets. The engine doesn't change a line.

The Textual TUI (--tui) is a 3-pane HUD: unit roster with live status lights, a streaming pursuit log, and the scenario summary. The pursuit runs in a worker thread; updates marshal back to the UI thread via call_from_thread.

 CYBERDYNE // SKYNET TERMINAL ────────── OBJECTIVE: John Connor
 UNITS              │ PURSUIT LOG                 │ SCENARIO
 ◤ SKYNET  ◉ think  │ ◤ SKYNET [opus-4-8]         │ target   John Connor
   opus-4-8         │   ANALYSIS ▸ …              │ alias    John Reese
 ❖ T-1000  ○ idle   │   DECISION ▸ deploy T-800   │ protector Sarah Connor
   sonnet-4-6       │ ⚙ T-800 deployed            │ relation mother
 ⚙ T-800   ● deploy │   ├ query_grid(Connor)      │

The lesson: program the engine against an interface, not a printer. Presentation becomes swappable ~ ANSI today, Textual now, a web UI or a MUD later, all over the same agent loop.

8 · Tech stack

Deliberately dependency-light: two runtime dependencies (anthropic, pydantic) ~ everything else is the Python standard library. No TUI framework, no web server, no database.

Language & runtime

Python 3.11+

Stdlib does the plumbing: argparse (CLI), tomllib (config), json, pathlib, dataclasses, time/sys (the theater).

uv

Environment + runner. uv run python -m skynet resolves and runs in one step.

hatchling

Build backend behind the pyproject.toml ~ makes skynet an installable package.

AI & agents

Anthropic Python SDK

messages.create(tools=…) drives each unit's agentic loop; messages.parse(output_format=…) returns validated structured output; thinking={"type":"adaptive"} on the reasoning tiers.

Claude models

claude-opus-4-8 (Skynet) · claude-sonnet-4-6 (T-1000) · claude-haiku-4-5 (T-800). Three tiers, one tradeoff.

Data & validation

Pydantic v2

Typed schemas for every inter-agent message ~ IntelReport and SkynetDecision. The structured-output contract.

TOML + JSON

config.toml holds the mission and the model/colour mapping; the synthetic grid is a flat .json file ~ no DB needed.

Presentation & testing

Raw ANSI truecolor

The CLI BBS theater (boot sequence, HUD, gradient bands, typewriter) is hand-rolled escape codes ~ no framework.

Textual + Rich

The interactive TUI front end (--tui): a 3-pane HUD over the same headless engine via the TextualUI adapter.

pytest

Smoke tests lock the grid's obfuscation, the randomiser's solvability, and the adapter drive-through ~ 14 green.

The worker's manual agentic loop

The heart of orchestrator.py. A manual loop (rather than the SDK tool-runner) keeps every step visible ~ which is the point of a teaching demo.

for _ in range(MAX_UNIT_STEPS):
    resp = client.messages.create(
        model=unit.model, system=unit.system_prompt,
        tools=grid.TOOL_SCHEMAS, messages=messages,
        max_tokens=8000,            # thinking= added for Sonnet/Opus, omitted for Haiku
    )
    messages.append({"role": "assistant", "content": resp.content})

    if resp.stop_reason != "tool_use":
        break

    tool_results = []
    for block in resp.content:
        if block.type == "tool_use":
            out = grid.execute_tool(db, block.name, dict(block.input))
            tool_results.append({"type": "tool_result",
                                 "tool_use_id": block.id, "content": out})
    messages.append({"role": "user", "content": tool_results})

# …then distil the work into a validated report
report = client.messages.parse(model=unit.model, messages=messages + [...],
                              output_format=IntelReport).parsed_output

11 · Key takeaways

  • Orchestrator ≠ worker. One agent reasons over results; others execute against tools. Keep them separate and the system stays legible.
  • Program to an interface, not a printer. The engine calls an injected ui; the CLI prints ANSI and the TUI posts to widgets over the same methods. Presentation became swappable for free.
  • Make the reasoning a first-class output. Assessment → reasoning → expectation as structured fields ~ the demo explains itself, and the data is the same in every front end.
  • Model tier is a design lever. Opus where reasoning is the bottleneck; Haiku where you want cheap volume. Costume aside, that's the real decision.
  • Structured output is the glue. Pydantic schemas between agents beat string-parsing ~ validated, typed, and refactor-safe.
  • Adversarial data makes it agentic. Obfuscate the target and a single query can't win; the agent has to reason. Without that, it's just a database.
  • Ship an offline mode. A deterministic --simulate path means the demo never fails in front of an audience, and the tests never hit the network.
